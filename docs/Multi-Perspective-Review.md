# Mr.IF Spark 多视角评审报告

> **评审对象**：MrIF-Spark-Product-Deck.md、system-prompt.md、signal-decode.ts、insight-engine.ts、reality-check.ts
> **日期**：2026-02-07

---

## 1. 普通用户视角（对金融有好奇心、无专业背景）

### 优势

1. **Setup → Flip → Land 叙事结构**：产品 Deck 示例「生病的人不买药，他们买的是无聊的解药」——先讲共识（药企），再翻转（流感季药企几乎没超额），最后落地（居家刷手机→流媒体 DAU）。这种结构容易产生「嗯？这个有意思」的体验。
2. **低门槛表达**：系统提示要求「金融术语保持英文」，输出控制在 ≤400 字，每条连接约 100–150 字，阅读负担小。
3. **诚实说「没什么」**：Worked Example 2 对「周一了情绪低迷」直接说「金融层面的信息量很低」，并建议「最好的操作可能就是不操作」，能降低被忽悠感。

### 不足与风险

1. **历史案例门槛**：insight_engine 输出如「2017-18 年 CDC 连续 19 周高于基线」「BLS 缺勤率 3.0%」——普通用户可能不知道 CDC/BLS 是什么，需要增加简短解释或链接。
2. **验证信号难落地**：如「关注 CDC ILINet 本周数据」——普通用户不知道去哪查、怎么解读，容易停在「有意思但没法用」。
3. **缺少「下一步」引导**：产品强调「然后你自己去查了」是价值所在，但未提供「查什么」「去哪查」的明确路径，可能削弱二次使用意愿。

### 关键问题

> 用户看完一条连接后，能否在 5 分钟内完成一次可执行的验证？如果不能，会不会觉得「有趣但没用」？

## 2. 专业投资者视角（CFA / 对冲基金分析师）

### 优势

1. **共识陷阱显式标注**：insight_engine 的 `consensus_traps` 明确列出「药企在流感季几乎没有超额表现」「天气预测已 priced in」等，有助于避免一阶共识。
2. **强制反例与做空者攻击**：reality_check 要求「MANDATORY: Also search for COUNTER-EXAMPLES」和 Gate 3 的「aggressive short-seller's attack」，与专业投资者的思维习惯一致。
3. **置信度分层**：HIGH（直接先例）/ MEDIUM（机制类比）/ LOW（纯推理）及对应语言，有助于区分可执行 vs 启发式思考。

### 不足与风险

1. **历史数据硬编码且可能过时**：insight_engine 中如「2017-18 Netflix Q1 subscriber beat +17%」「2022-23 Netflix LOST subscribers」等为静态数据。simulation-test-v1.md 已指出「2017 年 Netflix 数据可能已经不够新」。系统提示虽有 Step 2.5 网络检索，但 Spark MCP 本身未提供该工具，依赖客户端。
2. **reality_check 不执行验证**：reality_check 只输出验证框架（表格、模板），实际因果检验、历史验证、做空攻击均由 LLM 自行完成，存在「框架在但执行弱」的风险。
3. **部分量化表述需核实**：如「10Y +100bp → QQQ ~-8 to -12%」「DXY +10% → S&P EPS drag ~4-5%」——专业投资者会追问来源和样本期，目前无引用。

### 关键问题

> 若用户按 Spark 的「验证信号」去查 CDC/BLS 等数据，发现与 Spark 引用的历史案例不一致，会如何影响信任？是否有版本化、可追溯的数据来源？

## 3. 产品经理视角

### 优势

1. **差异化清晰**：产品 Deck 对比表（传统工具 vs ChatGPT vs Spark）中，「说没什么信号」「每条都标注弱点」是明确卖点，与「共识即陷阱」的定位一致。
2. **自适应深度**：signal_decode 的 magnitude（micro/medium/macro）与 insight_engine 的 depth guidance 结合，能根据输入复杂度调整输出深度，有利于留存。
3. **可扩展架构**：12 个 Skill 为独立 MD 文件，EVENT_CONFIGS 按事件类型组织，新增事件类型主要改配置，无需大改代码。

### 不足与风险

1. **留存路径不清晰**：没有会话历史、收藏、反馈闭环。产品 Deck 的 v2.0 路线图提到「用户标记『这条有意思』/『这条没用』」，但 v1.1 未实现，难以形成数据驱动的迭代。
2. **变现路径模糊**：定位为「思维起跑器」而非投资建议，规避合规的同时也限制了订阅、付费研报等变现方式，需要更明确的商业化假设。
3. **网络检索工具依赖外部**：index.ts 和 system-prompt 都依赖「网络检索工具」，但 Spark MCP 未注册该工具。若客户端未提供，事实验证和 Step 2.5 无法执行，会削弱产品承诺。

### 关键问题

> 用户在第 1 次、第 5 次、第 20 次使用时的价值差异是什么？是否有清晰的「越用越准/越有用」的机制？

## 4. 合规/法务视角（SEC / FINRA）

### 优势

1. **多重免责设计**：系统提示明确「NOT a financial advisor」「NOT giving investment advice」「NO specific buy/sell recommendations」，输出只到 theme/sector，ticker 仅作示例。
2. **无概率数字**：禁止「60% 概率」等表述，用定性置信度，降低「预测」色彩。
3. **诚实说「没什么」**：能主动说「这件事金融信号不强」「最好的操作可能就是不操作」，有助于论证「不是投资建议」。

### 不足与风险

1. **示例性 ticker 的边界**：如「代表性标的如 NFLX、DIS」——在监管语境下，列出具体标的可能被视为「暗示性推荐」。若用户追问「NFLX 和 DIS 哪个更好」，边界会进一步模糊。
2. **「验证信号」的解读**：如「关注 CDC ILINet 本周数据」——若用户理解为「按这个信号操作」，可能被视为隐含的投资指引。建议在输出中明确「这些信号是供你自行验证的，不是操作建议」。
3. **事实核查缺失**：系统提示要求对特定事实进行验证，但 Spark 依赖外部「网络检索工具」。若客户端不提供，LLM 可能基于未验证信息输出，存在误导风险。

### 关键问题

> 若用户因按 Spark 的「思考方向」操作而亏损，能否在诉讼/监管调查中清晰论证「Spark 从未给出投资建议」？每条输出的 disclaimer 是否足够显眼、可追溯？

## 5. 竞品分析师视角（vs ChatGPT / Perplexity / Bloomberg）

### 优势

1. **结构化金融推理链**：6 种桥接 × 正/逆、12 种事件类型、每类 1–3 条 chain 模板，形成可复用的推理路径。ChatGPT 和 Perplexity 多为通用对话，缺乏这种金融专用结构。
2. **对抗性验证机制**：4 道 Gate + 模式单调检查 + 强制逆向，是区别于通用 AI 的明显特征。Bloomberg 侧重数据与研报，不强调「攻击自己的假设」。
3. **共识陷阱与反常识**：consensus_traps、inversion_prompts 等显式设计，能系统性地避免「人人都想得到的角度」，ChatGPT 默认倾向共识性回答。

### 不足与风险

1. **历史数据与实时性**：ChatGPT/Perplexity 可检索最新信息；Bloomberg 有实时数据。Spark 依赖 insight_engine 的硬编码历史案例，若未接入实时数据，在 2022–23 流感季等反例上会落后。
2. **易被复制**：核心逻辑是「事件类型 → 链模板 → 共识陷阱 → 验证框架」，可被提炼为 prompt + 结构化配置。大厂可在通用模型中注入类似框架，快速做出「Spark 风格」产品。
3. **护城河在数据与反馈**：真正难复制的是：经用户反馈校准的链模板、实时验证的数据源、以及「这条连接后来被市场验证」的闭环。v1.1 尚未建立这些能力。

### 关键问题

> 6 个月后，若 OpenAI/Anthropic 推出「金融思维助手」并采用类似框架，Spark 还能靠什么保持差异？是数据、反馈闭环，还是与 Mr.IF 主产品的深度整合？

## 6. 技术架构师视角

### 优势

1. **MCP 工具链清晰**：signal_decode → insight_engine → reality_check 的调用顺序明确，职责划分清楚，便于调试和扩展。
2. **Skill 与代码解耦**：12 个 Skill 为独立 MD，通过 MCP resource 加载，修改推理逻辑无需改 TypeScript，符合「配置驱动」思路。
3. **Zod 校验**：工具参数使用 Zod schema，有利于类型安全和错误处理。

### 不足与风险

1. **reality_check 仅为框架**：reality_check 返回的是验证框架（表格、检查清单），不执行任何计算或检索。因果检验、历史验证、做空攻击全部依赖 LLM 自我执行，容易出现「有框架无约束」的情况。可考虑：对候选连接做结构化输出，由独立模块执行 Gate 1–2 的自动化检查。
2. **signal_decode 分类过于简单**：`classifyEvent` 仅用关键词匹配，`assessMagnitude` 用简单规则（如「我」「今天」→ micro）。对「最近好多人感冒了」这类混合信号，可能误判。建议在无法确定时，将 event_type 作为可选参数交由 LLM 覆盖。
3. **LLM 依赖集中**：整个流程由 LLM 串联，若模型幻觉或忽略工具输出，会直接影响质量。可考虑：对关键步骤（如历史案例引用）做格式校验或事后抽查。

### 关键问题

> reality_check 的 4 道 Gate 若改为「可执行代码」（如因果图检查、历史数据查询 API），能在多大程度上提升输出可靠性？实现成本与收益如何权衡？

## 综合评估

### 总体评分：7.5 / 10

| 维度 | 评价 | 影响 |
|------|------|------|
| 定位与哲学 | 清晰、有辨识度 | +1.5 |
| 金融逻辑 | 结构化强，但数据与验证有缺口 | -0.5 |
| 合规设计 | 基础到位，细节可加强 | -0.5 |
| 技术实现 | 架构合理，reality_check 偏弱 | -0.5 |
| 竞品差异 | 有特色，但护城河尚不稳固 | -0.5 |

---

### 上线前最需解决的 3 个问题

1. **网络检索工具依赖** — system prompt 多次提到"网络检索工具"，但 Spark MCP 本身没有注册这个工具。如果客户端不提供，事实验证和 Step 2.5 无法执行。需在 Spark MCP 内实现或明确约定接口；若客户端不提供，应在输出中明确标注「以上历史数据来自内置库，建议自行验证」。

2. **reality_check 执行力度** — 当前只提供框架（表格、模板），不做任何计算或校验。至少对 Gate 1（因果检验）和 Gate 2（历史验证）做部分自动化——例如要求 LLM 输出结构化字段（因果链、历史案例、反例），由代码做格式与完整性校验。

3. **合规 disclaimer 显眼度** — 在每条输出的固定位置（如开头或结尾）加入统一、可识别的免责声明，并考虑在日志中记录，便于事后审计。

---

### 当前最突出的 3 个差异化优势

1. **「诚实说没什么」** — 能主动承认低信号并建议不操作，与「共识即陷阱」的定位一致，在金融类 AI 中少见。

2. **对抗性验证框架** — 4 道 Gate + 做空攻击 + 强制反例，系统性地约束 LLM，降低 Beardstown Ladies 式自我欺骗。

3. **Setup → Flip → Land 叙事** — 把「共识 → 翻转 → 落地」固化为输出格式，有利于形成可识别的「Spark 风格」和用户心智。
